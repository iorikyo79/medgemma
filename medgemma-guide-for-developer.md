# MedGemma 모델 테스트 및 활용 가이드

* **작성자**: Ai Core LAB
* **사용 대상자**: 레포트팀, EI팀, Cardiology팀

이 문서는 배포된 MedGemma 모델 서버를 활용하여 레포트팀에서 다양한 실험(프롬프트 엔지니어링, 판독문 분석, 성능 평가 등)을 진행하고, 추후 파인튜닝(Fine-Tuning) 여부를 결정하기 위한 가이드라인입니다.

---

## 1. 개요 및 실험 목적

현재 사내 A100 서버에 **MedGemma-4B** 베이스 모델이 API 서버 형태로 구동되어 있습니다. 이 서버를 통해 다음과 같은 사전 타당성 검토(Feasibility Test)를 진행합니다.

1. **사전 학습된(Pre-trained) 모델의 한계 파악**: 베이스 모델이 우리의 도메인 지식(사내 판독문 스타일)을 얼마나 이해하고 있는지 확인.
2. **프롬프트 엔지니어링 한계 테스트**: 단순 명령(Zero-shot) 혹은 예시 제공(Few-shot)만으로 원하는 논리 교정이 가능한지 테스트.
3. **파인튜닝 필요성 판별**: 만약 프롬프트 튜닝만으로 성능이 부족하거나 일관성이 떨어진다면 구체적인 파인튜닝 데이터셋 구축 계획 수립.

## 2. API 엔드포인트 기본 사용법

배포된 서버는 REST 기반의 HTTP API를 제공합니다. Python 코드나 `curl` 등을 통해 내부 서버로 직접 질의할 수 있습니다.

* **엔드포인트 URL**: 
  * `http://10.10.40.194:8080/predict` (기본 제공)
  * `http://10.10.40.194:8080/v1/chat/completions` (OpenAI 표준 호환용)
  *(💡 참고: 두 엔드포인트는 내부적으로 완전히 동일한 엔진으로 연결됩니다. OpenAI 규격을 요구하는 서드파티 툴이나 라이브러리를 사용할 경우 `v1/chat/completions`를 사용하시면 됩니다.)*
* **메서드**: `POST`
* **Content-Type**: `application/json`

### 2.1. 기본 요청 구조 (JSON)

요청 데이터 포맷은 OpenAI의 Chat Completion 형태와 매우 유사합니다.

```json
{
  "messages": [
    {
      "role": "user", 
      "content": "이곳에 판독문을 포함한 분석 프롬프트를 입력합니다."
    }
  ],
  "max_tokens": 1024,
  "temperature": 0.1,
  "top_p": 0.9
}
```

* **`max_tokens`**: 모델이 생성할 수 있는 최대 단어(토큰) 개수입니다. 답변이 잘리지 않도록 충분히(예: 1024 이상) 설정하세요.
* **`temperature`**: 답변의 창의성(0.0 ~ 1.0). **의료/판독문 분석처럼 사실 기반의 정확한 판단이 필요한 경우 `0.1 ~ 0.2` 수준으로 아주 낮게 설정하는 것을 권장합니다.**
* **`top_p`**: 토큰 샘플링 확률(0.0 ~ 1.0). Temperature와 함께 생성의 다양성을 조절합니다. 일반적으로 `0.9`를 사용합니다.

## 3. 테스트 방법 (Python 및 cURL)

실험을 자동화하거나 터미널에서 빠르게 요청을 보낼 때 아래 두 가지 방법을 사용할 수 있습니다.

### 3.1. cURL 명령어를 이용한 빠른 테스트 (터미널)

터미널이나 커맨드 창에서 `curl` 명령어를 사용하여 모델을 즉각적으로 테스트할 수 있습니다.

```bash
curl -s -X POST http://10.10.40.194:8080/predict \
     -H "Content-Type: application/json" \
     -d '{
       "messages": [
         {
           "role": "user",
           "content": "다음 흉부 X-ray 판독문에 오탈자나 모순이 있는지 확인해 줘. [판독문: Both lugn fielsd에 특이 소견 안보임.]"
         }
       ],
       "max_tokens": 1024,
       "temperature": 0.1,
       "top_p": 0.95
     }' | jq -r '.choices[0].message.content'
```
*(💡 팁: 명령어 끝에 `| jq -r '.choices[0].message.content'`를 붙여주면 복잡한 JSON 형식은 무시하고, 한글로 변환된 최종 텍스트 답변 내용만 터미널에 깔끔하게 출력됩니다.)*

### 3.2. Python 테스트 스크립트 작성 예시 (자동화/배치 처리용)

실험을 엑셀 등 데이터셋과 연동하여 자동화하려면 Python의 `requests` 라이브러리를 사용하는 것이 좋습니다.

```python
import requests
import json

SERVER_URL = "http://10.10.40.194:8080/predict"  # A100 서버 API 주소

def analyze_report(prompt_text):
    payload = {
        "messages": [
            {"role": "user", "content": prompt_text}
        ],
        "max_tokens": 1024,
        "temperature": 0.1,  # 일관된 결과를 위해 온도를 낮춤
        "top_p": 0.95
    }
    
    headers = {"Content-Type": "application/json"}
    
    # 서버로 요청 전송
    response = requests.post(SERVER_URL, headers=headers, json=payload)
    
    if response.status_code == 200:
        result = response.json()
        # 답변 내용만 추출
        assistant_reply = result['choices'][0]['message']['content']
        return assistant_reply
    else:
        print(f"Error {response.status_code}: {response.text}")
        return None

# 테스트 실행
if __name__ == "__main__":
    prompt = "다음 판독문을 분석해줘: [판독문 내용]"
    print(analyze_report(prompt))
```

## 4. 유즈케이스별 프롬프트 실험 (Prompt Engineering)

다양한 판독문 시나리오에서 모델의 인지 능력을 평가하기 위한 프롬프트 전략입니다.
프롬프트를 짤 때는 모델 역할(Persona)과 작업(Task)을 아주 구체적으로 지시하는 것이 좋습니다.

### 유즈케이스 A: 판독문 내 오탈자 및 문법 오류 감지
의무기록사가 잘못 타이핑한 용어나 문맥상 어색한 단어를 찾아냅니다.

**프롬프트 예시:**
> 당신은 전문 영상의학과 전문의이자 꼼꼼한 의무기록 검수자입니다.
> 아래 제공된 영상의학 판독문에서 명백한 오탈자, 의학 용어의 철자 오류, 또는 문법적으로 어색한 부분을 찾아내고 올바른 단어를 제안해 주세요.
> 
> [판독문]
> 환자는 좌측 가슴 통을 호소하여 흉부 X선을 촬영함.
> Both lugn fielsd에 특이 소견 안보임.
> 심장 크기 정사범위임.
> 
> [출력 형식]
> 1. 원본 단어: xxx -> 수정 단어: yyy (이유 설명)

### 유즈케이스 B: 논리적 및 의학적 모순 감지 🚨 (핵심 테스트)
좌우 반전, 성별 모순, 이미 절제된 장기를 언급하는 등 로직상 불가능한 서술을 찾아냅니다.

**프롬프트 예시:**
> 당신은 20년 차 베테랑 임상 의사입니다.
> 다음 판독문에는 논리적으로 모순되는 내용이나 해부학적으로 성립할 수 없는 오류가 포함되어 있을 수 있습니다. 앞뒤 문맥을 비교하여 모순점(예: 우측을 검사했는데 좌측 병변이 나왔다거나, 남성 환자인데 자궁을 언급함 등)을 정확히 지적해 주세요. 문제가 없다면 '모순 없음'이라고 답변하세요.
> 
> [환자 정보]
> 성별: M, 나이: 45세, 과거력: 우측 신장 절제술(Rt. nephrectomy)
> 
> [판독문]
> Abdomen CT 소견. Liver, spleen 정상.
> 양측 신장(Both kidneys) 크기 및 모양 정상이며 수신증 없음.
> 전립선 비대 소견 관찰됨.
> 
> [분석 결과]:

### 유즈케이스 C: 복잡한 판독문의 요약 및 진단(Impression) 도출
수많은 Findings를 바탕으로 핵심 결론(Impression)을 요약할 수 있는지 테스트합니다.

**프롬프트 예시:**
> 아래는 흉부 CT의 소견(Findings)입니다. 이를 바탕으로 영상의학과 전문의가 작성하는 임프레션(Impression) 느낌으로, 가장 중요한 2~3줄의 결론만 짧게 요약해 주세요.
> 
> [CT Findings]
> (긴 소견 내용 기입...)
> 
> [Impression]

## 5. 성능 평가 및 파인튜닝 의사결정 로직

개발자 및 연구팀은 위 유즈케이스들을 직접 테스트한 후, 결과를 스프레드시트 등에 기록하여 아래 기준에 따라 다음 스텝을 결정합니다.

### 5.1. 베이스 모델의 한계 판별표

| 테스트 결과 (현상) | 분석 및 향후 대응 계획 (모델 튜닝) |
| :--- | :--- |
| 프롬프트 지시를 잘 따르고 80~90% 확률로 논리적 모순/오탈자를 완벽하게 짚어낸다. | **Zero-shot/Few-shot 프롬프트 엔지니어링으로 충분함.** 파인튜닝 불필요. 프롬프트 템플릿만 정교하게 다듬어서 앱에 탑재. |
| 의무 기록은 이해하나, 우리 병원만의 특수한 약어(Abbreviation)나 사투리(?) 같은 특정 텍스트를 인지하지 못한다. | **도메인 특화 약어집 RAG(검색 증강) 추가 적용** (파인튜닝 전 RAG 먼저 고려) 혹은 소규모 **LoRA 파인튜닝**으로 용어 주입. |
| 결과가 너무 들쭉날쭉하다. (어떨 땐 맞추고 어떨 땐 헛소리(Hallucination)를 창조한다) | **Instruction Fine-tuning (지시어 미세조정) 필수.** "이런 형태의 데이터가 들어오면, 무조건 이런 포맷으로 답변해라" 라는 Question-Answer 데이터셋(1천 개 이상)을 구축해야 함. |
| 겉으로 보이는 문장은 그럴싸한데, 의학적인 로직 추론 능력 자체가 아예 결여되어 있다. (예: 폐렴과 골절 수치를 구분 못함) | MedGemma 4B 모델의 기본 체급 한계일 수 있음. 더 큰 모델(예: MedGemma 27B) 도입 고려 또는 파인튜닝 데이터의 난이도를 낮춰 아주 작은 단위의 태스크로 모델 분할. |

### 5.2. 개발팀 To-Do List

1.  **테스트 데이터 수집**: 실제 임상에서 사용 중인 과거 판독문 50~100건(정상 케이스 + 고의로 오류를 주입한 케이스 혼합). (⚠️ 개인정보 완전 비식별화 필수)
2.  **배치 테스트 스트립트 구동**: 100건의 데이터를 위 Python 스크립트와 연동하여 자동으로 돌리고 응답을 수집.
3.  **정량 평가 (Accuracy)**: 모델이 짚어낸 에러 중, 실제로 의사가 인정하는 'True Positive'의 비율과 'False Alarm' 비율 산출.
